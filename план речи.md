# Язык «Речь» — стартовый каркас

## Идея
«Речь» — это язык программирования с русским синтаксисом, где программа читается как последовательность обычных предложений. Цель проекта: компиляция или интерпретация через собственный рантайм, написанный на ассемблере x86-64.

## Принципы
- Синтаксис на русском языке.
- Инструкции выглядят как короткие предложения.
- Минимум символов, максимум смысла.
- Сначала интерпретатор, потом компилятор.
- Целевая платформа: x86-64 Linux, NASM.

## Минимальный набор конструкций

### Присваивание
```text
пусть x будет 10.
пусть имя будет "Аня".
```

### Условие
```text
если x больше 5, то
    скажи x.
иначе
    скажи 0.
конец.
```

### Цикл
```text
пока x меньше 10, делай
    пусть x будет x плюс 1.
конец.
```

### Вывод
```text
скажи "Привет".
```

### Возврат
```text
верни x.
```

### Типы токенов
- IDENT — имя переменной
- NUMBER — целое число
- STRING — строка в кавычках
- KEYWORD — ключевое слово
- OP — оператор сравнения или арифметики
- EOL — конец строки / точка
- INDENT / DEDENT — отступы

## Грамматика первого уровня

```text
program      := statement*
statement    := assign | if_stmt | while_stmt | print_stmt | return_stmt
assign       := 'пусть' IDENT 'будет' expr '.'
print_stmt   := 'скажи' expr '.'
return_stmt  := 'верни' expr '.'
if_stmt      := 'если' expr ',' 'то' block ('иначе' block)? 'конец.'
while_stmt   := 'пока' expr ',' 'делай' block 'конец.'
block        := statement*
expr         := comparison
comparison   := sum (('равно' | 'не' | 'больше' | 'меньше' | 'или' | 'и') sum)*
sum          := term (('плюс' | 'минус') term)*
term         := factor (('умножить' | 'разделить') factor)*
factor       := NUMBER | STRING | IDENT | '(' expr ')'
```

## Как хранить программу внутри интерпретатора

Для простого старта достаточно такого представления:

```text
struct Token {
    type
    value
    line
    column
}

struct Node {
    kind
    left
    right
    value
}
```

Но в ассемблере лучше упростить и хранить:
- тип токена — в байте,
- смещение строки в исходнике — в dword/qword,
- длину лексемы — в dword,
- указатель на данные — в qword.

## Архитектура интерпретатора на NASM

### Файлы проекта
- `main.asm` — входная точка
- `lexer.asm` — разбор текста на токены
- `parser.asm` — сборка AST или простого списка команд
- `runtime.asm` — выполнение команд
- `strings.asm` — работа со строками
- `io.asm` — вывод в stdout
- `memory.asm` — простая память под узлы и токены

### Этапы работы
1. Прочитать файл с кодом языка.
2. Разбить текст на токены.
3. Построить список команд.
4. Исполнить команды.

## Первый практический вариант: не полноценный AST, а линейный исполнитель

Для первого релиза лучше не строить сложное дерево. Достаточно распознавать такие шаблоны:
- `пусть X будет N.`
- `скажи X.`
- `если ... то ... иначе ... конец.`
- `пока ... делай ... конец.`

Это ускорит разработку.

## Каркас вывода

```asm
; io.asm
BITS 64

GLOBAL print

SECTION .text

; print(rdi = addr, rsi = len)
print:
    mov rax, 1      ; sys_write
    mov rdx, rsi
    mov rsi, rdi
    mov rdi, 1      ; stdout
    syscall
    ret
```

## Каркас чтения файла

```asm
; file_read.asm — очень грубый задел
BITS 64

GLOBAL read_file

SECTION .text

; read_file(rdi = fd, rsi = buffer, rdx = max_len) -> rax = bytes_read
read_file:
    mov rax, 0      ; sys_read
    syscall
    ret
```

## Лексер: первое правило

На первом шаге лексер должен уметь:
- пропускать пробелы и табы,
- выделять слова,
- выделять числа,
- выделять строки в двойных кавычках,
- выделять точку как конец команды.

### Пример токенизации
Вход:
```text
пусть x будет 10.
```

Выход:
- KEYWORD(`пусть`)
- IDENT(`x`)
- KEYWORD(`будет`)
- NUMBER(`10`)
- EOL(`.`)

## Предлагаемый формат ошибок

```text
Строка 3, символ 12: ожидалось слово «будет».
Строка 8: неизвестное имя переменной «хуйня».
```
